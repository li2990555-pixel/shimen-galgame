## 一、产品概述

### 1.1 一句话描述

> 写一段通勤路上的观察，AI会让你知道——有人和你在同一节车厢里想过类似的事。

### 1.2 产品简介

这是一个基于"流动空间"的轻量日记体平台。用户记录自己在通勤途中（地铁、路口、便利店等）的观察或心情，AI在用户已收藏的故事之间自动发现连接，生成"如果这些陌生人相遇了"的虚构对话。

产品形态为移动端适配的web应用，3天内由AI代码生成工具（Bolt.new/v0）完成构建。

### 1.3 核心逻辑

1. 用户以匿名身份撰写日记，标记所在空间（如"2号线"）
2. 以Tinder式卡片浏览他人日记，右划收藏到个人素材库
3. AI在素材库内扫描日记之间的空间/主题连接
4. 对匹配的日记对，AI生成一段"如果相遇"的虚构对话
5. 用户在素材库中查看收集的故事和AI生成的相遇对话

## 二、问题与机会

### 2.1 背景

人类学家项飙提出"附近的消失"——城市人越来越只关注自身（个体经验）和远方（宏大意识形态），中间的附近被跳过。但现有关于"附近"的讨论默认其锚定在**定居空间**（小区、邻里、菜市场），而忽略了另一个维度：

**流动空间**——地铁、路口、便利店、电梯——这些我们每天置身其中、与大量陌生人共处但从不交流的空间，也是一种"附近"。它们没有消失，只是从未被激活。

### 2.2 问题

| 问题 | 描述 |
|------|------|
| 流动空间中存在大量未被激活的"附近" | 每天和几百个陌生人共处一车厢，但没有任何机制让这种共存被感知 |
| 城市人缺乏轻量的表达出口 | 不是不想写，是没有一个"低门槛、有回应"的场景 |
| 现有的陌生人社交工具都基于身份 | Tinder基于外貌，Soul基于标签，没有一个工具围绕"故事的连接"展开 |

### 2.3 机会

| 机会 | 说明 |
|------|------|
| 市场空白 | 已有大量产品做"家族口述史传承"，但无产品专注于"记录附近陌生人" |
| AI成熟 | AI的文本理解和生成能力使得"发现故事之间的隐性连接"成为可能 |
| 低门槛 | 用户只需写一段通勤观察，不需要专业创作能力 |
| 情感价值 | "原来有人和我看到了一样的事"——这种共时性体验是强烈的情感触发点 |

## 三、目标用户

### 3.1 核心用户画像

| 属性 | 描述 |
|------|------|
| **身份** | 一二线城市通勤族，18-35岁 |
| **通勤方式** | 地铁/公交/步行 |
| **媒介习惯** | 日均手机使用>5小时，习惯碎片化内容消费 |
| **情感状态** | 对城市生活有疏离感，有观察欲但缺少出口 |
| **技术素养** | 熟练使用手机应用，对新产品接受度高 |

### 3.2 用户场景

| 场景 | 触发 | 行为 | 情绪 |
|------|------|------|------|
| 通勤路上 | 看到车厢里的某个人/某件事 | 打开产品，写一段观察 | 好奇→记录→期待 |
| 睡前 | 回顾一天 | 刷卡片，收集故事 | 放松→共鸣 |
| 收到通知 | AI发现了素材库内的连接 | 打开素材库阅读相遇对话 | 惊喜 |
| 第二天通勤 | 想起昨天的体验 | 继续写，继续收集 | 形成了习惯 |

## 四、产品定位

### 4.1 差异化定位

| 对比维度 | 家族口述史产品 | 陌生人社交产品 | **本产品** |
|---------|--------------|--------------|-----------|
| 核心关系 | 血缘 | 身份/外貌 | **故事** |
| 内容形式 | 长篇口述 | 照片/标签 | **微日记** |
| 连接方式 | 存档/传承 | 匹配/聊天 | **AI发现** |
| 情感基调 | 怀旧 | 猎奇 | **共鸣** |
| 时间导向 | 过去 | 现在 | **共时性** |

### 4.2 产品价值观

- **不暴露身份**：匿名是安全表达的前提
- **制造相遇**：不是"你主动去找人"，而是"AI让你知道你并不孤单"
- **克制**：不设计聊天功能、不推交友、不做个性化推荐算法

### 4.3 产品名称候选

| 名称 | 理由 | 状态 |
|------|------|------|
| 擦肩 | 擦肩而过——两个陌生人最接近的瞬间 | 待定 |
| 同车厢 | 直接描述使用场景，有代入感 | 待定 |
| 途中 | 在路上，在途中。简洁开放 | 待定 |

## 五、功能需求

### 5.1 功能全景

| 模块 | 功能 | 优先级 | 复杂度 | 说明 |
|------|------|--------|--------|------|
| 日记 | 写日记 | P0 | 低 | 文字输入 + 空间标记 |
| 日记 | 匿名昵称生成 | P0 | 低 | 随机生成（如"栗子乘客"） |
| 浏览 | 卡片流 | P0 | 中 | Tinder式堆叠 + 左右划手势 |
| 浏览 | 左划跳过 | P0 | 低 | 卡片滑出，不保存 |
| 素材库 | 右划收藏 | P0 | 低 | 日记进入个人素材库 |
| 素材库 | 素材库列表 | P0 | 低 | 已收集日记的列表视图 |
| AI连接 | 连接扫描 | P0 | 高 | 素材库内日记的空间/主题匹配 |
| AI连接 | 相遇对话生成 | P0 | 高 | AI根据匹配结果生成对话 |
| AI连接 | 对话展示 | P0 | 中 | 素材库中的对话卡片展示 |
| 浏览 | 卡片展开详情 | P1 | 低 | 点击查看完整日记内容 |
| 日记 | 语音输入 | P1 | 中 | 语音转文字后提交 |
| 浏览 | 空间筛选 | P1 | 低 | 按空间标签过滤卡片 |
| 浏览 | 已读去重 | P1 | 中 | 已划过的卡片不再重复出现 |

### 5.2 功能详情

#### 5.2.1 写日记（P0）

**前置条件**：无（匿名即可写，无需登录）

**主流程**：
1. 用户点击"写日记"按钮
2. 进入写日记页，显示文字输入区
3. 用户输入日记正文（字数不限）
4. 选择/输入空间标签（预置选项：各城市地铁线 + "其他"自定义输入）
5. 系统自动生成匿名昵称（每次随机）
6. 点击"发布"
7. 日记存入数据库，进入卡片池，用户回到卡片流

**验收标准**：
- 用户可在30秒内完成一篇日记发布
- 空间标签至少支持预置5个选项 + 自定义输入
- 匿名昵称每次不同，不与用户身份关联

#### 5.2.2 卡片流（P0）

**前置条件**：卡片池中有至少1条日记

**主流程**：
1. 进入首页，显示堆叠式卡片
2. 顶部卡片完整展示日记内容+空间标签+匿名昵称
3. 用户左划 → 卡片向左滑出 → 下一张卡片出现（不保存）
4. 用户右划 → 卡片向右滑出 → 进入素材库 + 震动反馈 → 下一张卡片出现
5. 用户点击卡片 → 展开详情（如已开启AI连接，显示相遇对话）

**验收标准**：
- 手势流畅无卡顿
- 左划/右划有明确的视觉反馈
- 右划后底部素材库计数立即+1
- 无日记时显示空状态引导："还没有人来过这里，写下第一条通勤观察吧"

#### 5.2.3 素材库（P0）

**前置条件**：用户至少右划了1条日记

**主流程**：
1. 用户点击底部"素材库"入口
2. 显示已收集故事总数 + AI相遇对话（如有）
3. AI相遇对话卡片：如素材库内有匹配的日记对，展示生成的对话
4. 下方为已收集日记的列表，按收藏时间倒序

**验收标准**：
- 素材库数据存储在localStorage（匿名，无需用户系统）
- AI连接在后台异步生成，生成后素材库显示标记
- 用户可删除已收藏的日记

#### 5.2.4 AI相遇对话生成（P0）

详见[第九节：AI提示词设计](#九ai提示词设计)

## 六、非功能需求

### 6.1 性能

| 指标 | 目标 |
|------|------|
| 页面加载时间 | <2秒（首屏） |
| 卡片滑动帧率 | >55fps |
| AI对话生成延迟 | <10秒（异步，后台生成） |
| 并发支持 | >100同时在线用户 |

### 6.2 安全

| 要求 | 说明 |
|------|------|
| 无用户系统 | 不收集手机号/邮箱/微信等个人信息 |
| 匿名默认 | 所有用户以随机昵称出现，不可追踪真实身份 |
| 内容审核 | MVP阶段不设自动审核，部署时加基本敏感词过滤 |

### 6.3 可用性

| 要求 | 说明 |
|------|------|
| 移动端优先 | 核心体验为手机竖屏设计 |
| 触屏优化 | 手势滑动响应区域>60px |
| 桌面可用 | 桌面端鼠标拖拽替代手指滑动 |

## 七、数据模型

### 7.1 日记（entries）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | uuid | PK | 自动生成 |
| content | text | NOT NULL | 日记正文 |
| space | varchar(50) | NOT NULL | 空间标签 |
| space_custom | varchar(100) | nullable | 自定义空间名 |
| author_name | varchar(30) | NOT NULL | 匿名昵称 |
| device_id | varchar(100) | NOT NULL | 素材库归属标识 |
| created_at | timestamptz | DEFAULT now() | 创建时间 |

### 7.2 AI连接（connections）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | uuid | PK | 自动生成 |
| entry_a_id | uuid | FK → entries.id | 日记A |
| entry_b_id | uuid | FK → entries.id | 日记B |
| dialogue | text | NOT NULL | AI生成的相遇对话 |
| summary | varchar(200) | nullable | 连接原因摘要 |
| device_id | varchar(100) | NOT NULL | 归属于哪个用户的素材库 |
| created_at | timestamptz | DEFAULT now() | 生成时间 |

### 7.3 索引

- entries: `(created_at DESC)` — 卡片流排序
- entries: `(space)` — 空间筛选
- connections: `(device_id)` — 按用户素材库查询

### 7.4 存储实现

- **数据库**：Supabase PostgreSQL
- **匿名标识**：localStorage 中存储 `device_id`（UUID v4），用于标识用户而不需登录
- **存储策略**：日记公开可读，连接按 device_id 归属

## 八、交互流程

### 8.1 核心流程

```
[打开应用]
     │
     ▼
[首页：卡片流] ←←←←←←←←←←←←←←←←←←←←
     │                                    ↑
     ├── 左划 → 跳过，下一张 ←→ 无日记？→ 空状态引导写日记
     │                                    ↑
     ├── 右划 → 收藏到素材库 + 计数+1 → 下一张 ←───┘
     │                                    ↑
     ├── 点击卡片 → 展开详情 + AI对话（如有）───┘
     │
     ├── [✏️] 写日记 → 输入内容 + 选空间 → 发布 → 回到卡片流
     │
     └── [📚] 素材库 → 浏览收集的故事 + AI相遇对话
                         │
                         └── 点击AI对话 → 查看原始日记
```

### 8.2 异常流程

| 场景 | 处理方式 |
|------|---------|
| 卡片池为空 | 显示空状态："还没有人来过这里，写下第一条通勤观察吧" |
| AI生成超时 | 对话区域显示"正在生成中..."，下次打开时完成 |
| 素材库为空 | 显示引导："去首页收集你感兴趣的故事吧" |
| 网络断开 | 显示"网络连接断开"，本地已缓存的内容可继续浏览 |
| 提交内容为空 | 发布按钮置灰，提示"写点什么吧" |

## 九、AI提示词设计

### 9.1 核心提示词

```
你是一个城市观察者。以下两条日记来自同一个空间（{space}），
由不同的人在不同时间写下。

日记A：{contentA}
日记B：{contentB}

请生成一段80-150字的虚构对话，场景设定为：
如果这两个人在{space}中擦肩而过、对视了一眼。

对话要求：
1. 自然、日常、不煽情
2. 从两篇日记的共同点出发（比如都注意到了同一件事）
3. 保持克制，不要过度戏剧化
4. 对话结束后仍保持陌生人的距离感
5. 使用中文口语，不超过150字

只输出对话本身，不要加任何说明文字。
```

### 9.2 连接发现逻辑

| 步骤 | 说明 |
|------|------|
| 触发条件 | 用户素材库新增日记（N≥2时） |
| 扫描范围 | 仅限该用户素材库内的全部日记 |
| 匹配规则 | space相同 或 主题语义相似（调用AI判断） |
| 去重 | 已生成连接的日记对不再重复生成 |
| 存储 | 连接记录存入connections表 |
| 展示 | 素材库中按时间倒序展示连接对话 |

### 9.3 API调用方式

- **模型选择**：GPT-4o-mini / DeepSeek-V3 等性价比模型
- **调用策略**：批量异步调用，不阻塞前端
- **频率限制**：每次触发最多扫描10个候选对
- **回退方案**：AI调用失败时，素材库展示"正在寻找连接"占位

## 十一、路演方案

### 11.1 6分钟路演结构

| 时长 | 内容 |
|------|------|
| 0:00-1:00 | **问题陈述**：地铁上每天有几百人和你擦肩而过，但你不知道他们中任何一个人的名字。这不是冷漠，是没有机制。 |

| 5:00-6:00 | **未来方向**：语音输入、更多的流动空间覆盖、跨城市的"同一时刻"连接 |
