## 一、产品概述

### 1.1 一句话描述

> 家政女工按住说话，AI帮你分类，同城同行匿名互助。

### 1.2 产品简介

「家政女工互助平台」是一个面向家政女工的语音互助社区。家政女工通过微信扫码进入，按住说话就能发布语音消息——AI自动将语音转为文字并打上分类标签（求助/避雷/分享/唠嗑），匿名发布到同城同行的信息流中。每个人都是"某某阿姨"，没有头像、没有个人主页、没有点赞评论，语音条48小时自动消失。

产品核心理念是"无社交压力的同行互助网络"——不复制微信的熟人社交模式，也不复制小红书的达人流量模式，而是创造一个纯粹的、匿名的、低门槛的同行信息交换空间。

### 1.3 核心逻辑

**信息流模式 + 语音驱动 + 匿名身份 + 自动消失**

- **入口**：老带新扫码 → 语音自我介绍 → AI自动生成"姓氏+随机名"的匿名身份
- **发布**：按住说话 → 松开 → AI转文字 + 自动分类标签 → 发布到同城同行圈
- **浏览**：按分类标签筛选浏览 → 点一下听原声 → 可看AI生成的文字摘要
- **消亡**：每条语音48小时后自动删除，不留痕迹
- **验证**：发布前需完成从业验证（录制一段与家政工作相关的语音，AI判断相关性）

## 二、问题与机会

### 2.1 背景

家政女工是中国城市中规模庞大却高度"隐形"的群体。据《阿姨年鉴2024》数据，中国家政服务从业人员已超过3000万人，其中85%以上为女性，平均年龄46岁。她们每天穿梭在城市的高楼大厦之间，却极少被城市公共空间所容纳。

在数字空间中，她们的处境同样边缘化。据中国社会科学院2023年调研，家政女工智能手机普及率已达96%，但APP安装数量平均仅为3.2个——微信几乎构成了她们的全部数字生活。

### 2.2 核心问题

**问题一：微信群信息过载，有价值的内容被淹没。**

家政女工已经在微信群里自发了互助网络——互相推荐靠谱工作、曝光黑心雇主、分享收纳技巧、倾诉委屈。但微信群是线性聊天，信息杂乱无结构。一条重要的"XX小区雇主拖欠工资"的避雷信息，可能在50条"早上好"表情包之后被彻底淹没。新人进群没有任何入口可以回顾历史信息。

**问题二：打字门槛高，语音是最自然的表达方式。**

据《阿姨年鉴2025》数据，家政女工中初中及以下学历占53%，很多人在打字和文字阅读上存在困难。但她们普遍熟练使用微信语音功能——语音是她们最自然的数字表达方式。现有互助平台（58到家、阿姨帮等）以文字和图片为主，没有为语音交互做过优化。

**问题三：个人信息安全焦虑，害怕被雇主看到。**

CHI2025家政工隐私研究（Zhao et al., 2025）表明，家政女工对数字隐私有高度焦虑——她们害怕自己在网上说的内容被雇主看到导致丢工作、被投诉。绿色蔷薇深圳家政工调研（2024）进一步指出，家政女工最需要的数字空间特征是"雇主不在场"——她们需要一个同行专属的、雇主无法进入的私密空间。

**问题四：现有平台不匹配她们的互助需求。**

现有的家政平台（58到家、天鹅到家等）是"雇主-阿姨"匹配平台，核心功能是简历展示和接单——这些平台强化了雇主对阿姨的审视关系，而非同行之间的互助关系。微信群虽是互助场所，但结构过于松散，不具备信息沉淀和检索能力。

### 2.3 机会

1. **刚需明确**：家政女工已经在自发做这件事（微信群互助），说明需求真实存在，产品只是提供一个更好的工具
2. **竞争空白**：市场上没有专门面向家政女工的语音互助社区，这是一个明确的产品空白地带
3. **AI降门槛**：语音转文字+自动分类技术已经成熟，可以大幅降低信息整理和检索的成本
4. **隐私红利**：匿名+自动消失的设计，可以精准回应家政女工的隐私焦虑——这是她们愿意尝试新产品的核心驱动力
5. **网络效应**：同城互助场景天然具有网络效应——一个城市的人越多，信息越有价值

### 3.2 典型场景

**场景一：避雷——发现黑心雇主**

张姐在雇主家工作了三天，雇主以"试用期不合格"为由拒绝支付工资。张姐想提醒同行注意这个小区这个雇主。她打开「家政女工互助平台」，按住说话："XX花园8栋的黄太太，说好试用期三天给500，今天说我不合格一分钱不给，姐妹们注意避雷"。松开后，AI自动生成文字并在每条前面加上标签 [避雷]。同城的姐妹刷到这条语音，点一下听到张姐的原声，确认了信息。

**场景二：求助——急找今天的工作**

李阿姨今天被临时通知不用来，一天没收入很着急。她按住说话："姐妹们，我今天在龙华这边空出来了，有没有哪家需要临时做保洁的？我可以做半天"。AI分类为[求助]，发布到同城圈。半小时后有同行回复（通过发布新语音@她的方式），推荐了一个临时缺人的雇主。

**场景三：分享——技能互助**

王姐学会了一个快速去除油烟机油污的方法，按住说话："用食用碱+热水泡抹布，盖在油污上十分钟，一擦就掉，比什么清洁剂都好用"。AI分类为[分享]，其他姐妹在浏览时可以点一下听语音，也可以看AI转的文字。

**场景四：唠嗑——情感慰藉**

晚上10点干完活，刘姐一个人在雇主家的保姆间里，觉得孤单。打开「家政女工互助平台」说："今天东家的小孩叫我'阿姨'，我说叫阿姨就行，别叫'保姆阿姨'，听着心里不舒服。"AI分类为[唠嗑]。过了一会儿听到几条同城姐妹的语音回复："理解你""我也遇到过""我们就是阿姨，不是什么保姆"。

## 四、产品定位

### 4.1 差异化定位

**「家政女工互助平台」不是……**
- 不是58到家：不提供接单和雇主匹配服务，只做同行互助
- 不是微信：不是熟人社交，是匿名同行社区
- 不是小红书：没有达人/粉丝/流量逻辑，每个人都是平等的"阿姨"
- 不是知乎/贴吧：不依赖文字生产能力，语音是第一表达方式

**「家政女工互助平台」是……**
> 家政女工的"语音版同行互助社区"——低门槛、匿名、自动消失、雇主免入。

### 4.2 产品价值观

**价值观一：无雇主**

这是家政女工自己的空间。通过从业语音验证确保雇主无法进入。产品设计上不设置任何雇主入口、不提供雇主视角的功能、不在任何公开渠道暴露"家政"相关关键词。

**价值观二：无社交压力**

没有粉丝数、没有点赞、没有评论、没有个人主页、没有排行榜。每个人只有一个"某某阿姨"的匿名身份。发布内容不追求流量，只追求对同行有用。

**价值观三：低门槛优先**

语音驱动的核心交互方式，降低文字表达门槛。UI设计大字体、大按钮、高对比度，适配中老年人的使用习惯。关键操作都有语音引导。

**价值观四：同级互助**

每个人都是"阿姨"，没有等级制度、没有大V、没有KOL。信息价值由内容本身决定，而非发布者的身份。验证机制确保"阿姨"身份的真实性（确实是在做家政的人）。

**价值观五：隐私是默认的**

不需要手机号注册，无需登录。语音条48小时自动消失。不收集任何个人信息。不展示位置精确信息（只展示到城市级别）。

### 5.2 功能详情

#### 5.2.1 注册与验证

**功能描述：**

这是一个两步验证的注册流程。

第一步，新用户通过老用户分享的二维码扫码进入。页面显示欢迎语和简短的语音引导。

第二步，用户录制一段与家政工作相关的语音（例如"我叫XX，做家政三年了，主要在福田做保洁"）。AI判断语音内容是否与家政工作相关，若不相关则引导重新录制。通过后，AI从语音中提取姓氏（"我姓张"→"张"或从自我介绍中提取），与随机配名（如"秀丽""美华"等常见中老年女性名字用字）组合，生成"张阿姨""王阿姨"等匿名身份。

**验收标准：**
- 扫码后首次访问，显示注册页而非内容页
- 能成功录制至少10秒的语音
- AI对"我做了三年钟点工"等正例判断通过率>90%
- AI对"今天天气真好"等负例判断拒绝率>80%
- 成功生成"姓+阿姨"格式的显示名
- 同一设备再次访问直接进入内容页（信息保存在localStorage或IndexedDB）

#### 5.2.2 语音发布

**功能描述：**

页面底部有一个大按钮（占屏幕宽度60%以上），文字提示"按住说话"。用户按住按钮开始录音，松开发送到服务器。发送后，AI依次执行：
1. 语音转文字（ASR）
2. 内容分类（[求助]/[避雷]/[分享]/[唠嗑]）
3. 将语音文件、转写文字、分类标签一起存储

发布过程中显示加载状态（"正在处理…"），处理完成后语音自动出现在信息流顶部。用户可听自己的语音回放确认。

**验收标准：**
- 按住按钮开始录音，松开结束录音，延迟<300ms
- 录音最长60秒，最短3秒（太短提示"多说两句吧"）
- 语音转文字准确率>85%（针对带口音的普通话）
- 分类标签准确率>80%
- 从松手到发布完成显示在信息流中，总耗时<10秒
- 支持在录音中途上滑取消发送

#### 5.2.3 语音浏览

**功能描述：**

信息流是核心浏览页面。默认展示同城（同一城市）的所有语音条目，按发布时间倒序排列。顶部有分类标签栏（全部/求助/避雷/分享/唠嗑），点击筛选。

每条语音条目展示：
- 用户昵称（如"张阿姨"）—— 灰色小字，弱化个人身份
- 分类标签（彩色小标签：[求助]为红色、[避雷]为橙色、[分享]为绿色、[唠嗑]为蓝色）
- 语音转写的文字内容（2-3行预览，点击可展开全文）
- 发布时间（如"2小时前""昨天"）
- 播放按钮（点击播放原声）

点击条目播放语音时，自动高亮对应的文字内容（类似卡拉OK歌词效果），帮助用户同步听和看。

**验收标准：**
- 首次进入无内容时显示空状态引导页（示例内容+引导发第一条）
- 标签筛选切换流畅，无感知延迟
- 点击语音条目开始播放，再次点击暂停
- 播放时对应文字高亮
- 列表支持虚拟滚动，200条内容无明显卡顿
- 城市自动识别（基于IP或浏览器地理位置API）

#### 5.2.4 隐私与消失机制

**功能描述：**

这是产品的核心信任机制。

每条语音发布时记录时间戳。系统的定时任务（每5分钟运行一次）检查所有语音条目，删除发布时间超过48小时的记录（包括语音文件、转写文字、分类标签）。用户无法主动删除自己的语音（降低被雇主要求删帖的压力："不是我删的，是系统自动没了的"）。

不需要手机号注册，不使用任何持久性登录态。使用localStorage存储"阿姨"身份ID。清除浏览器缓存等于"注销"——但这不是问题，因为语音会自然消失，身份本身没有累积价值。

**验收标准：**
- 语音条在48小时整自动从信息流消失
- 自动删除作业对用户无感知（不提示"XX已删除"）
- 用户退出页面重新进入无需重新登录
- 清除浏览器缓存后重新进入，需要重新注册
- 服务器端没有任何持久化的用户身份信息（仅有匿名ID）

### 5.3 不做（明确不包含的功能）

- ❌ 不做私信/一对一聊天
- ❌ 不做点赞/评论/转发
- ❌ 不做个人主页/头像/个人介绍
- ❌ 不做雇主入口/雇主版本
- ❌ 不做接单/交易/支付功能
- ❌ 不做粉丝/关注/好友关系
- ❌ 不做信息永久存档/搜索历史
- ❌ 不做Push通知推送（MVP阶段）

## 六、非功能需求

### 6.1 性能需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 页面加载时间 | <3秒（首屏） | 考虑目标用户使用中低端安卓手机 |
| 语音上传时间 | <5秒（30秒录音） | 语音文件需压缩优化 |
| 语音转文字延迟 | <3秒 | 服务端处理时间 |
| 页面交互响应 | <100ms | 按钮点击、标签切换 |
| 并发用户数 | MVP初期支持1000人同时在线 | 初期用户量级预估 |
| CDN加载 | 按需配置 | 语音文件使用CDN加速 |

### 6.2 安全需求

| 需求 | 说明 |
|------|------|
| 语音文件访问控制 | 仅已验证用户可访问，无法直接通过URL分享 |
| 从业验证门槛 | 语音内容必须与家政工作相关，防止雇主混入 |
| 无手机号/无登录 | 不采集任何PII（个人身份信息） |
| XSS防护 | 用户生成的文字内容转义后展示 |
| 反滥用 | 同一设备/IP短时间内大量发布触发限流（1分钟内最多发3条） |
| 语音内容审核基础 | 初步关键词过滤（暴力、色情等），MVP可使用开源方案 |

### 6.3 可用性需求

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 主键 |
| `anonymous_id` | VARCHAR(32) | 匿名身份标识（非用户ID，仅用于去重和关联） |

**索引：**
- `idx_city_created` ON (`city`, `created_at` DESC)

- `idx_expires_at` ON (`expires_at`) WHERE `is_deleted` = false
- `idx_anonymous_id` ON (`anonymous_id`)

#### 7.1.2 阿姨身份表（classmates）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 主键 |
| `anonymous_id` | VARCHAR(32) | 匿名身份标识（随机生成） |
| `surname` | VARCHAR(4) | 从语音提取的姓氏 |

| `last_active_at` | TIMESTAMPTZ | 最后活跃时间 |

**说明：** 此表不包含手机号、真实姓名、地址等任何PII。`device_fingerprint`仅用于同一设备的身份恢复，通过浏览器指纹生成，不可反向解析。

| key | label | color | sort_order | description |

| help | 求助 | #FF4444 | 1 | 急找工作、需要帮助、询问信息 |
| alert | 避雷 | #FF8C00 | 2 | 曝光黑心雇主、提醒风险、投诉 |
| share | 分享 | #4CAF50 | 3 | 技能技巧、经验分享、好雇主推荐 |

### 7.2 不存储的数据

- ❌ 手机号
- ❌ 真实姓名
- ❌ 地址/定位（仅存到城市级别）
- ❌ 雇主家庭信息
- ❌ 登录密码
- ❌ 第三方账号信息

### 7.3 Supabase配置说明

- **Database**: PostgreSQL，启用 Row Level Security (RLS)
- **Storage**: 用于存储语音文件，设置公开访问但限制referrer
- **RLS策略**:
  - `voice_posts`：SELECT 允许所有已验证用户；INSERT 需要验证token；DELETE 仅限定时任务（service_role）
  - `voice_posts`中已过期数据的清理：使用Supabase的pg_cron定时任务
  - `classmates`：INSERT 仅限注册流程；SELECT 不允许（前端不直接查询此表）

### 8.2 异常流程

| 从业验证不通过 | 提示示例："可以说说您做家政的经历，比如'我做保洁三年了'" |

| 存储空间满（语音文件） | 返回用户友好的错误提示，异步通知管理员 |

## 九、AI设计

### 9.1 语音转文字（ASR）

**技术选型建议：**

| 方案 | 优势 | 劣势 |
|------|------|------|
| 阿里云语音识别 | 对口音普通话识别率较高，中文优化 | 收费，开通需企业认证 |
| 腾讯云语音识别 | 中文场景优化好，微信生态配合度高 | 收费 |
| Deepgram | 高精度，多语言支持 | 可能对中文口音支持不如国内方案 |
| OpenAI Whisper API | 准确率高，对口音鲁棒性好 | 境外API，延迟可能较高 |
| 本地Whisper部署 | 免费，数据不外传 | 需要GPU资源，MVP阶段不推荐 |

**MVP推荐方案：** 阿里云或腾讯云语音识别（国内API，延迟低，中文优化好）。如果出于成本考虑，可以先使用OpenAI Whisper API做快速原型，后续迁移到国内方案。

**参数要求：**
- 采样率：16kHz（平衡质量与文件大小）
- 音频格式：WebM/Opus（浏览器原生支持，压缩率高）
- 语言：中文普通话，启用方言自适应

### 9.2 分类标签Prompt设计

**系统提示词：**

```
你是一个内容分类助手。你的任务是将家政女工的语音转写文本分类到以下四个类别之一。

分类规则：
- [求助]：用户正在寻求帮助、询问信息、急找工作。包含"有没有""谁知道""怎么办""求介绍""帮帮我"等关键词。
- [避雷]：用户正在曝光不良雇主、提醒风险、投诉维权。包含"黑心""拖欠""不给钱""避雷""小心""注意"等关键词。
- [分享]：用户正在分享经验、技巧、好雇主推荐。包含"推荐""分享""教你们""技巧""好用""方法"等关键词。
- [唠嗑]：用户正在闲聊、倾诉心情、日常交流。包含表达情绪、生活琐事、与同行打招呼等无明显求助/避雷/分享意图的内容。

输出格式：仅输出分类标签（help/alert/share/chat），不要输出其他内容。

示例：
输入："姐妹们，XX花园8栋的黄太太，试用期三天不给钱，大家注意避雷"
输出：alert

输入："请问龙华这边有没有今天需要临时做保洁的？"
输出：help

输入："用食用碱加热水泡抹布，盖在油污上十分钟，一擦就掉"
输出：share

输入："今天东家的小孩叫我阿姨，心里挺不是滋味的"
输出：chat
```

**分类准确率提升策略：**

1. 对于置信度<0.7的case，默认分类为[唠嗑]（最安全的选择），前端允许用户手动修改
2. 定期收集人工修正的数据，微调few-shot示例
3. 针对家政业务场景的特殊词汇（"东家""阿姨""钟点工"等），建立行业词库辅助分类

### 9.4 从业验证Prompt设计

**系统提示词：**

```
你是一个从业验证助手。判断用户语音转写文本是否与家政服务工作相关。
```

## 十一、路演方案

### 11.1 路演核心叙事（3分钟版）

**开场（30秒）：家政女工的"隐形"困境**

"全中国有3000万家政女工，她们每天在城市的家里进进出出，但没有人看到她们。她们用微信群互相帮忙——推荐工作、曝光黑雇主、分享技能——但微信群里的信息就像流水一样，发完就找不到了。"

**问题切入（30秒）：为什么微信群不够用？**

"她们中的53%只有初中以下学历，打字很困难，但所有人都用微信语音。微信群一天几百条消息，重要的避雷信息被'早上好'的表情包淹没了。而且她们最怕的是——被雇主看到自己在说什么，可能丢工作。"

**解决方案（60秒）：「家政女工互助平台」是什么？**

"「家政女工互助平台」是一个语音互助社区——按住说话，AI帮你转成文字，自动分类打标签。求助、避雷、分享、唠嗑，一目了然。每个人都是'张阿姨''王阿姨'，没有头像、没有粉丝、没有评论。语音48小时自动消失，不需要手机号。最重要的是——我们通过语音验证确保雇主进不来。"

**演示（40秒）：现场操作**

"扫码进入→发一段语音自我介绍→AI提取姓氏，生成'张阿姨'→按住说话→松开→AI转文字+贴标签→发布到同城同行圈→刷到别人的语音，点一下听。"

**收尾（20秒）：我们的愿景**

"家政女工需要的不是被城市看见，她们需要的是被同行看见。我们不为她们创造流量，我们只是给她们的互助网络配上一个更好的工具。"

### 11.2 现场演示脚本

**准备内容：**
1. 一个已注册好的测试账号（名为"张阿姨"）

## 十二、开放问题

1. **语音文件存储成本：** 每条语音60秒、按16kHz/16bit算，单条约480KB。1000个用户每人每天发3条，30天约43GB。MVP阶段使用Supabase免费额度（1GB Storage + 2GB带宽）够用约2天。后续需考虑成本控制：压缩码率、缩短最长时长、或设置每日发布上限。

2. **从业验证的准确性：** 用AI判断语音是否与家政相关，在MVP阶段可能有10-20%的误判率。验证过松则雇主混入，过严则阻碍真实用户进入。需要在Launch后快速收集数据并优化。

3. **城市冷启动问题：** 如果一个城市只有几个人用，信息流会很空。初期推广策略应该集中火力在一个城市（建议深圳，绿色蔷薇已有社区基础），形成密度后再扩展。

4. **音频内容审核：** 纯语音内容比文字更难做审核。MPV阶段使用转写后的文字做关键词过滤，但无法过滤语音中的语气、情绪等信息。长期需要更完善的内容审核机制。

5. **"阿姨"身份的防伪：** localStorage + 设备指纹的方案在更换设备或清除缓存后会丢失身份。这是一个设计选择——"身份不重要，内容才重要"。但如果用户反馈强烈，后续可考虑通过微信OpenID（不获取用户信息，仅作为身份恢复凭据）来做轻量级身份恢复。

6. **合规风险：** 不需要手机号注册在中国法律灰色地带。建议咨询律师确认ICP备案和《个人信息保护法》合规要求。MVP阶段在上线前至少完成必要的法律文本准备（隐私政策、用户协议）。
